promise学习总结

发表于 2015-06-23   |   分类于 javascript

端午节唯一的收获，就是由jquery源码，知道了Promise大概是个啥。ES6的学习有了一个不错的开始。话不多说，先上一个简易代码，过几天弄懂之后再补充一段专业而严谨的解说。
我对Promise的直观看法：node.js官方的api基本都是以会回调方式传递函数返回值，层层嵌套，不知不觉就建造起了一座高高的回调金字塔。针对这种普遍问题，Promise应势而生！

简单demo

function timeout(ms){
    return new Promise(function(resolve,reject){
        setTimeout(function(){
            resolve('貌似要成功了哦');
            /*reject();*/
        },ms);
    });
}
timeout(1000).then(function(v){
    alert(v);
},function(){
    alert('竟然失败了额')
});

timeout对象返回一个promise。代码的执行效果很简单，1000ms之后，弹出成功语句；或者是注释resolve，解放出reject，此时会弹出失败语句。

var p1=new Promise(function(resolve){
        setTimeout(function(){
            resolve('hello');
        },1000);
    });
var p2=new Promise(function(resolve){
    setTimeout(function(){
        resolve('world');
    },2000);
});
Promise.all([p1,p2]).then(function(result){
    alert(result);
});

Promise.all 原理及使用场景

Promise.all 可以接收一个元素为 Promise 对象的数组作为参数，当这个数组里面所有的 Promise 对象都变为 resolve 时，该方法才会返回。
上面的例子模拟了传输两个数据需要不同的时长，虽然p1的速度比p2要快，但是 Promise.all 方法会按照数组里面的顺序将结果返回。
使用场景一：从不同接口拉取数据，数据到达的时间不同，但是需要把所有接口的数据合并处理在返回；使用场景二：当需要等一个耗时很长的for循环完全结束才调用下一个函数时候，可用Promise.all包装这个循环。

对Ajax的封装，以及抛出错误时的另一种处理方式：

function get(url){
    return new Promise(function(resolve,reject){
        var req=new XMLHttpRequest();
        req.open('get',url,true);
        req.onload=function(){
            if(req.status==200){
                resolve(req.responseText);
            }else{
                reject(new Error(req.statusText));
            }
        }
        req.error=function(){
            reject(new Error('Network error'));
        }
    });
}
var url='XXXX';
get(url).then(function(v){
    console.log(v);
}).catch(function(e){
    console.log(e);
});

我推荐使用catch而不是then(null,dealError(e))的形式处理错误。在于catch只需要写一次，然后只要有错误产生就会抛出。而dealError必须在每一个then函数里面重复。

异步的体现：

var promise=new Promise(function(resolve){
    console.log('inner promise'); //1 最先显示
    resolve(42);
});
promise.then(function(v){
    console.log(v); //3  最后显示
});
console.log('outer promise');  //2 然后显示

javascript是按照出现顺序从上往下执行的，所以1先出现没有异议；关键是2和3的顺序问题。1执行之后，紧跟着返回resolve对象，promise对象翻转为“确认”状态，然后给promise注册了then事件。请注意，此时then对象的回调函数是异步的！所以3先执行，等环境中没有执行代码之后，异步函数推入环境，执行完毕。

参数传递：同步返回

function A(v){
    return v*2;
}
function B(v){
    return v+1;
}
function C(v){
    alert(v);
}
var promise=Promise.resolve(100);
promise
    .then(A)
    .then(B)
    .then(C)
    .catch(function(e){
        console.log(e);
    });

tip：每个函数都有一个then函数，then函数内部的返回有三种形态：return一个promise，后一个then函数会等待返回后异步执行；return一个同步的值（不返回则是returnundefined），后一个函数会同步执行，此时then已经没有什么意义了；throw一个同步异常。

参数传递：promises穿透

Promise.resolve('foo').then(Promise.resolve('bar')).then(function (result) {
  console.log(result);
}); //结果输出foo

这个结果的原因是：如果你给then传递的参数不是函数，而是promise，这个then()会直接被忽略，导致前一个promise直接穿透。所以永远在then中传递函数，我的经验是最好传递匿名函数，再返回要执行的函数。

我来解释微博上传的很火的四道题目

doSomething，doSomethingElse，finalHandler都默认返回promise。

doSomething().then(function () {
  return doSomethingElse();
}).then(finalHandler);

正常执行。doSomething返回之后执行doSomethingElse，执行完之后再执行finalHandler，完全没有问题。

doSomething().then(function () {
  doSomethingElse();
}).then(finalHandler);

悲剧了。第一个then没有返回，则等同于返回undefined。所以两个then会同步执行。

doSomething().then(doSomethingElse())
  .then(finalHandler);

悲剧了。第一个函数传递的并不是函数，而是函数的返回值，即promise，所以产生了promises穿透！所以后一个then会跟谁doSomething的返回执行。doSomethingElse并没有错误，其会与doSomething并行执行。

doSomething().then(doSomethingElse)
  .then(finalHandler);

正常执行。then传递的均是函数句柄，与第一种情况相比，除了this不同，其余都是一样的额。
